# 2026-04-10

Summary: Orchestrator now routes through the sub-orchestrator with a dedicated `routing_memory` lane and a hard raw-message cutoff on that lane

> **TL;DR**: The sub-orchestrator is now wired to a dedicated `routing_memory` queue with a hard 5k cutoff; the remaining follow-up inside that lane is replacing prune-only behavior with real summarization/compression.

## Updates

### 2026-04-10 | Orchestrator routing-memory cutoff wire

- changed: removed the old orchestrator-side summarizer/check path and kept the main parser on full `messages`, while the sub-orchestrator now reads from a dedicated `routing_memory` message lane
- changed: converted `routing_memory` from a string field into `Annotated[list, add_messages]` so it behaves like a real message queue instead of a prompt-injected blob
- changed: both the latest human turn in `input_parser` and the latest AI turn in `output_compiler` now mirror into `routing_memory`
- changed: added `backend/message_window.py` and a hard queue-prune rule for `routing_memory`; when the lane crosses the 5k token budget, the append path drops the oldest 3/4 of the lane and keeps appending new turns
- pending: the lane still has no summarizer/compression logic; the next follow-up is replacing the hard prune with a real routing-memory summarizer once the rest of the sub-orchestrator token policy is ready
